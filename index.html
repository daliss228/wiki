🚀 Guía de Desarrollo

Flujo de Trabajo GitHub Enterprise

🎯

Propósito

Establecer un proceso estándar para la gestión del código fuente, garantizando trazabilidad, calidad y colaboración entre los miembros del equipo de desarrollo.

1

Estrategia de Ramas

📖 Descripción

La estrategia de ramas define cómo se organiza el desarrollo dentro de GitHub Enterprise.

🎯 Objetivo

Evitar conflictos entre desarrolladores y mantener estable la rama principal.

🌳 Estructura

main
develop
feature/*
hotfix/*
feature/login-biometrico
feature/prediccion-ahorro
hotfix/error-calculo-impuestos
2

Pull Requests y Revisiones

📖 Descripción

Todo cambio debe ser revisado antes de integrarse a la rama principal.

🎯 Objetivo

Garantizar calidad y compartir conocimiento dentro del equipo.

✅ Reglas

  • No fusionar directamente a main.
  • Mínimo una aprobación.
  • Validaciones automáticas aprobadas.
Pull Request
Implementación de módulo de recomendaciones inteligentes.
Revisor
Desarrollador Senior.
Resultado
Aprobado y fusionado a develop.
3

Convenciones de Commits

📖 Descripción

Los commits deben seguir una nomenclatura estándar.

🎯 Objetivo

Facilitar el seguimiento histórico de cambios.

📝 Formato

tipo: descripción
feat: agregar módulo de autenticación
fix: corregir cálculo de saldo disponible
docs: actualizar documentación de API
refactor: optimizar consulta de usuarios
4

Gestión de Versiones

📖 Descripción

Cada liberación debe registrarse mediante etiquetas.

🎯 Objetivo

Permitir rastrear fácilmente versiones desplegadas.

🏷️ Versiones

v1.0.0
v1.1.0
v1.1.1

📋 Registro de Liberación

  • Fecha de liberación.
  • Responsable.
  • Funcionalidades incluidas.
  • Problemas corregidos.
